本文档是给后续 AI 编程助手准备的 Android 方向接手文档。
目标不是解释“项目大概是什么”，而是让接手人可以直接继续推进 `android/` 主线，而不必重新猜当前状态、边界和坑点。

重要要求：

1. 后续 AI 编程助手不能只改代码不改文档；
2. 只要 Android 主线、真实联调方式、输入法路线、真机排查步骤发生变化，就必须同步更新：

   - [docs/安卓真机联调清单.md](/Users/sam/Downloads/typeink/docs/安卓真机联调清单.md)
3. 如果是继续维护输入法主线，请优先再读：
   - [docs/安卓输入法完整实现方案与测试计划.md](/Users/sam/Downloads/typeink/docs/安卓输入法完整实现方案与测试计划.md)

## 1. 先说结论

当前 Android 方向已经从“WebView 测试壳”切到“原生 Android 客户端 + 原生输入法服务”双入口。

当前主线判断：

1. Android 主路径应该继续走原生客户端与原生输入法，不再回到网页壳。
2. 当前 APK 已同时包含普通 App 入口和 `InputMethodService` 输入法入口。
3. 当前 Android 客户端负责：
   - 原生 UI
   - 原生录音
   - 原生本地实时草稿
   - DashScope 实时 ASR
   - DashScope 智能改写
   - 输入法基础编辑能力
   - “修最近一句”语音修正
4. 本仓早期 Python / Web 验证资产已归档移除，Android 主线不再依赖本仓服务端脚本。
5. 后续是否要正式发布、是否继续扩成更完整的键盘布局，应在现有原生输入法主线上继续收敛，而不是另起一条 WebView 路线。

一句话总结：

当前最正确的接续方式是“继续把原生 Android 输入法体验做稳”，而不是“回到网页壳”，也不是“重新发明一套并行客户端”。

### 3.4 DashScope 语音与改写链路

- [DashScopeService.kt](/Users/sam/Downloads/typeink/android/app/src/main/java/com/typeink/prototype/DashScopeService.kt)
- [TypeinkRewriteSupport.kt](/Users/sam/Downloads/typeink/android/app/src/main/java/com/typeink/prototype/TypeinkRewriteSupport.kt)

职责：

1. 建立 DashScope 实时 ASR WebSocket
2. 发送 PCM 音频流
3. 处理 `task-started / result-generated / task-finished`
4. 构造改写提示词
5. 解析 DashScope OpenAI 兼容 SSE 流式返回
6. 在模型失败时回退到可用文本，避免 UI 一直卡住

当前边界：

1. 这里现在已经是 Android 主线里的核心链路，不再只是轻量转发层
2. `TypeinkSessionClient.kt` 已经成为历史 WebSocket 方案遗留，不是当前主路径
3. 目前仍缺少更完整的断线重试和网络分类

### 3.5 原生音频采集

- [PcmRecorder.kt](/Users/sam/Downloads/typeink/android/app/src/main/java/com/typeink/prototype/PcmRecorder.kt)

职责：

1. 用 `AudioRecord` 采集 16kHz、单声道、PCM16 音频
2. 按 `640` 字节音频负载分块
3. 交给 `DashScopeService` 发给云端 ASR

注意：

1. 当前 Android 主线发送的是原始 PCM 分片，不再走旧的 opcode WebSocket 协议
2. 如果未来重新切回 Python 服务端代理，这里的输出协议需要重新审视

### 3.6 本地实时草稿层

- [LocalDraftRecognizer.kt](/Users/sam/Downloads/typeink/android/app/src/main/java/com/typeink/prototype/LocalDraftRecognizer.kt)

职责：

1. 用系统 `SpeechRecognizer` 提供 interim 草稿
2. 在云端 `asr_partial / asr_final` 到达前，先让 UI 有即时可见文本
3. 云端回流一到，就退出本地草稿层

当前风险：

1. 强依赖系统语音识别能力，不同手机差异很大
2. 某些机型可能同时使用 `AudioRecord` 与 `SpeechRecognizer` 时表现不稳定
3. 这里只是“体验兜底层”，不是最终的稳定实时识别方案

## 5. 后端协议必须知道什么

协议定义在：

- [app/schemas.py](/Users/sam/Downloads/typeink/app/schemas.py)

Android 客户端当前实际依赖的最小协议如下。

### 5.1 客户端 -> 服务端

1. `start_session`
   - 用来开始一轮会话
   - 当前 Android 客户端会把 `rewriter.style_mode` 一起带过去

2. `context_update`
   - 仅在“语音修改模式”使用
   - 传：
     - `selected_text`
     - `pre_text`
     - `post_text`

3. `stop_recording`
   - 结束录音后发送

4. 音频二进制帧
   - 第 1 个字节是 opcode
   - 后面是 PCM16 数据

### 5.2 服务端 -> 客户端

1. `session_ready`
2. `asr_partial`
3. `asr_final`
4. `llm_delta`
5. `llm_completed`

### 5.3 当前 Android 端对这些消息的处理

1. `asr_partial`
   - 写到 ASR 区
   - 普通输入模式下写回主文本框
   - 关闭本地实时草稿层

2. `asr_final`
   - 写到 ASR 区
   - 普通输入模式下写回主文本框
   - 关闭本地实时草稿层

3. `llm_delta`
   - 累积显示在 AI 区

4. `llm_completed`
   - 作为最终文本写回主文本框

## 6. 当前 Android 主线怎么启动

这部分已经和早期 Python / Web 联调方式不同了。

现在要优先记住一句话：

当前仓库里的 Android 主线，已经不再依赖本仓 Python 服务端脚本。

### 6.1 本地 Key 放在哪里

当前推荐放在：

- [android/local.properties](/Users/sam/Desktop/typeink-codex/android/local.properties)

在已有 `sdk.dir=...` 的基础上，追加：

```properties
typeink.dashscope.apiKey=你的DashScopeKey
```

注意：

1. `android/local.properties` 是本机私有文件，不进 Git；
2. 如果需要临时切换机器，也可以改用环境变量 `TYPEINK_DASHSCOPE_API_KEY` 或 `DASHSCOPE_API_KEY`；
3. 不要再把真实 Key 写回源码、README 或文档。

### 6.2 当前正确构建方式

```bash
cd /Users/sam/Desktop/typeink-codex/android
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" ./gradlew :app:assembleDebug
```

单元测试：

```bash
cd /Users/sam/Desktop/typeink-codex/android
JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home" ./gradlew :app:testDebugUnitTest
```

### 6.3 如果怀疑 Key 没生效

优先检查：

1. `android/local.properties` 里是否真的存在 `typeink.dashscope.apiKey`
2. Key 是否写在 `android/` 工程下，而不是继续只放在仓库根目录
3. 构建后 App 是否仍提示 `DashScope Key 还没配置`
4. 是否刚改完配置但还没重新构建 APK

### 6.4 早期服务端联调路径的状态

原来的 `ws://IP:8000/ws`、`8001` 上游 ASR、本仓 `uvicorn` 启动方式，现在都属于历史验证路径。

这条路对应的仓库资产已经被收缩移除，不再作为当前 Android 主线说明。

## 7. 当前已经踩过的坑

### 7.1 不要再把“移动版 App”默认理解成 WebView 壳

这个坑已经踩过一次了。

结论：

1. WebView 方案只适合最短路径试装
2. 不适合继续做 Android 主路线
3. 现在 `android/` 的主线已经切到原生客户端，不要再绕回去

### 7.2 如果看到“累计收到 xxx bytes 音频”

这不是 Android 的 UI 问题，是后端又跑在 `mock` 模式了。

请优先检查：

1. `TYPEINK_ASR_PROVIDER`
2. `TYPEINK_REWRITER_PROVIDER`
3. `android/local.properties` 中的 Key 是否真的被构建读取
4. 当前安装到手机上的是否是重新构建后的新 APK

### 7.3 “没有实时草稿”不一定全是 Android UI 的锅

当前实时草稿来自两层：

1. 本地 `SpeechRecognizer`
2. 云端 `asr_partial`

如果还是没实时出字，可能有三种原因：

1. 本地 `SpeechRecognizer` 在当前手机上不可用
2. 设备同时跑 `AudioRecord + SpeechRecognizer` 的表现不好
3. DashScope 实时回流不够快，客户端只能等句末

### 7.4 “为什么结果不够专业”

先确认当前风格模式是不是 `formal`。

因为：

1. `natural` 现在被定义成“修顺但不改味道”
2. `formal` 才是“更清晰、更专业、更适合发出去”

如果产品想要更强的“秘书整理稿”效果，应该继续加强 `formal` 提示词，而不是把 `natural` 变成公文腔。

## 8. 当前产品语义

### 8.1 `natural`

当前目标：

1. 去掉明显累赘
2. 去掉口头禅、重复和无意义噪声
3. 修顺病句
4. 保留本人语气

不应该做的事：

1. 不要把自然口气硬改成公文腔
2. 不要因为“看起来更专业”就把用户的个人表达全磨平

### 8.2 `formal`

当前目标：

1. 允许更清晰、更专业、更适合正式沟通
2. 可补结构、补逻辑连接、整理列表
3. 仍不能编造新信息

当前仍需继续优化：

1. 整理强度还不够“像专业秘书”
2. 还需要更多真机与真实语音样例来调 prompt

## 10. 当前不建议做的事

### 10.1 不建议把 Python 主服务硬塞进 APK

原因：

1. 当前主服务与上游 ASR 都是 Python 服务形态
2. 硬塞进 APK 会让包体、生命周期、稳定性和调试复杂度大幅上升
3. 这不是当前最值钱的投入点

正确路线是：

1. 先把 Android 客户端做稳
2. 再决定哪些能力要迁移到端上

### 10.2 不建议现在立刻做 `InputMethodService`

原因：

1. 一旦切输入法服务，复杂度会明显上升
2. 现在普通 App 形态还没完全跑顺
3. 先把录音、状态机、重写体验做稳更值钱

## 11. 推荐的下一步优先级

如果后续 AI 助手要继续推进，推荐顺序如下：

### P1：把当前普通 App 形态做稳

优先做：

1. 真机验证本地实时草稿在不同机型上的表现
2. 给 `formal` 模式增强整理力度
3. 增加更细的错误提示和首字时延显示
4. 拆 `MainActivity`，引入更清晰的状态管理层

### P2：把客户端体验做得更像产品

优先做：

1. 更像 Typeless 的界面结构
2. 更清楚的“识别中 / 优化中 / 写回完成”反馈
3. 连接失败、识别失败、重写失败的兜底态

### P3：再单独规划输入法形态

这一阶段才考虑：

1. `InputMethodService`
2. 系统输入框写回
3. 选区读取
4. 宿主 App 兼容性

## 12. 如果下一位 AI 助手只能做一件事
