# 本地 ASR 与 VAD 升级实施计划

> **给后续实现者：** 先按本计划完成“接口收口 -> TEN-VAD -> sherpa-onnx -> 波形反馈”的顺序，不要把三类能力混成一次大改。`icon` 不在本计划范围内。

**Goal:** 在不打断当前 Android IME 主链路的前提下，把 Typeink 从“系统草稿识别 + 能量阈值停录”升级到“可回退的模型化 VAD + 可演进的本地 ASR 底座”。

**Architecture:** 先把草稿识别、VAD、波形反馈从当前具体实现里抽成稳定接口，保证现有 DashScope 路径仍可运行；再优先引入 `TEN-VAD` 替换判停层，因为它对真实交互收益最大、接入成本低于本地 ASR；之后再引入 `sherpa-onnx` 作为本地识别底座，最后才处理 `WaveLineView` 这类反馈层替换。

**Tech Stack:** Android 原生 Kotlin、InputMethodService、现有 `DashScopeService + PcmRecorder + AsrSessionManager`、[TEN-VAD](https://github.com/TEN-framework/ten-vad)、[sherpa-onnx](https://github.com/k2-fsa/sherpa-onnx)、[WaveLineView](https://github.com/Jay-Goo/WaveLineView)、Gradle、本地模型资源（AAR / so / onnx / model assets）。

---

## 一、结论先说

1. 第一优先级：`TEN-VAD`
   原因：它直接决定“停录准不准”，对当前体验提升最大，而且可以沿用现有 `VadProcessor` 接口渐进替换。
2. 第二优先级：`sherpa-onnx`
   原因：它是后续真正离线化的底座，但接入成本明显高于 VAD，涉及模型、AAR、资源组织、加载时机和包体。
3. 第三优先级：`WaveLineView`
   原因：它提升的是“录音时看起来更顺滑”，不是 P0。当前已有 `AudioWaveView` 可用，不必抢在识别内核前面接。

## 二、总体分阶段

### 阶段 A：先把接口收口，不急着换库

**目标：** 把 VAD、草稿识别、波形反馈从“具体实现”改成“可替换接口”，为后续渐进接入做准备。

**Files:**
- Create: `android/app/src/main/java/com/typeink/asr/DraftRecognizer.kt`
- Modify: `android/app/src/main/java/com/typeink/prototype/LocalDraftRecognizer.kt`
- Modify: `android/app/src/main/java/com/typeink/vad/VadProcessor.kt`
- Modify: `android/app/src/main/java/com/typeink/vad/VadConfig.kt`
- Modify: `android/app/src/main/java/com/typeink/inputmethod/AsrSessionManager.kt`
- Modify: `android/app/src/main/java/com/typeink/prototype/MainActivity.kt`
- Test: `android/app/src/test/java/com/typeink/vad/VadConfigTest.kt`

**步骤：**
1. 新建 `DraftRecognizer` 接口，抽出 `start / stop / isAvailable` 这一层，不让 `AsrSessionManager` 直接绑死系统 `SpeechRecognizer`。
2. 让现有 `LocalDraftRecognizer` 退化成 `SystemDraftRecognizer` 角色，先作为默认实现保留。
3. 扩展 `VadConfig`，把“当前 VAD 实现类型”做成显式配置，例如：`ENERGY`、`TEN`、`DISABLED`。
4. 保持 `AsrSessionManager` 只依赖接口，不感知底层是系统识别、sherpa 还是哪种 VAD。
5. 先补配置与默认值测试，确保旧链路不因抽象层改动而挂掉。

**完成标准：**
1. 不引入任何新模型时，现有录音、停录、远端 ASR、文本写回行为不回退。
2. `AsrSessionManager` 可以通过配置切换 VAD 与草稿识别实现。

### 阶段 B：优先接入 TEN-VAD，替换判停层

**目标：** 用 `TEN-VAD` 替换当前能量阈值停录主路径，同时保留 `EnergyVadProcessor` 作为兜底。

**Files:**
- Modify: `android/app/build.gradle`
- Create: `android/app/src/main/java/com/typeink/vad/TenVadProcessor.kt`
- Create: `android/app/src/main/java/com/typeink/vad/TenVadFactory.kt`
- Modify: `android/app/src/main/java/com/typeink/vad/VadConfig.kt`
- Modify: `android/app/src/main/java/com/typeink/inputmethod/AsrSessionManager.kt`
- Create: `android/app/src/main/assets/vad/ten-vad.onnx`
- Test: `android/app/src/test/java/com/typeink/vad/VadProcessorSelectionTest.kt`

**步骤：**
1. 先按 `参考目录` 的做法，把 `ten-vad.onnx` 放进 `assets/vad/`，不要先散落到别的目录。
2. 在 Gradle 里补齐 `TEN-VAD` 需要的 JNI / so 接入方式，优先选择“项目内固定版本”而不是运行时下载。
3. 新建 `TenVadProcessor`，让它实现现有 `VadProcessor` 接口，保持 `onSpeechStart / onSilenceDetected / onSpeechEnd` 语义不变。
4. 参考 参考键盘 的经验补上 3 个关键防抖：
   - 初始开口防抖；
   - 短静音挂起；
   - 实例池或预热，避免首次录音卡顿。
5. 默认策略改为：
   - 能加载 `TEN-VAD` 就走 `TEN`；
   - 加载失败自动回落到 `ENERGY`；
   - 设置页保留手动切换入口，方便排障。

**完成标准：**
1. 真机下“说一句后短暂停顿”不再像能量阈值那样容易误停或晚停。
2. `TEN-VAD` 加载失败时，不影响当前录音主流程。

### 阶段 C：接入 sherpa-onnx，替换草稿识别底座

**目标：** 把现有系统 `SpeechRecognizer` 草稿识别，替换为真正可控的本地 ASR 路径。

**Files:**
- Modify: `android/app/build.gradle`
- Create: `android/app/libs/`
- Create: `android/app/src/main/java/com/typeink/asr/sherpa/SherpaDraftRecognizer.kt`
- Create: `android/app/src/main/java/com/typeink/asr/sherpa/SherpaReflectionSupport.kt`
- Create: `android/app/src/main/java/com/typeink/asr/sherpa/SherpaModelConfig.kt`
- Modify: `android/app/src/main/java/com/typeink/inputmethod/AsrSessionManager.kt`
- Modify: `android/app/src/main/java/com/typeink/prototype/LocalDraftRecognizer.kt`
- Create: `android/app/src/main/assets/asr/`
- Create: `android/app/src/test/java/com/typeink/asr/SherpaModelConfigTest.kt`

**步骤：**
1. 先确定我们只接一条最小闭环模型路线，不一开始就支持 参考键盘 那种多本地模型矩阵。
   推荐起点：单个中文优先、包体可控、社区资料完整的 sherpa 模型。
2. 先把 `sherpa-onnx` 的 Android AAR 和相关 native 资源固定到工程中，避免构建时远程漂移。
3. 新建 `SherpaDraftRecognizer`，只承担“本地草稿转写”这一件事，不在第一版里同时做标点、后处理、热词、下载中心。
4. 复用 `DraftRecognizer` 接口，让 `AsrSessionManager` 只知道“这里有个本地草稿源”，不知道它是不是 sherpa。
5. 默认策略改为：
   - sherpa 模型可用时，优先给 IME 提供本地草稿；
   - sherpa 不可用时，自动回退到系统 `SpeechRecognizer`；
   - 再不行就只保留远端 ASR 最终结果。

**完成标准：**
1. 飞行模式下，草稿层仍能得到本地识别结果。
2. 不要求第一版完全离线润色，但要证明“本地转字”这层已经独立可控。

### 阶段 D：补设置、预热与模型管理，不让功能只停留在代码里

**目标：** 让 `TEN-VAD` 与 `sherpa-onnx` 不只是“代码可编译”，而是“用户能控制、系统能预热、问题可排查”。

**Files:**
- Modify: `android/app/src/main/java/com/typeink/settings/SettingsActivity.kt`
- Modify: `android/app/src/main/java/com/typeink/settings/ProviderListActivity.kt`
- Modify: `android/app/src/main/java/com/typeink/vad/VadSettingsActivity.kt`
- Create: `android/app/src/main/java/com/typeink/asr/LocalModelPreloadCoordinator.kt`
- Modify: `android/app/src/main/java/com/typeink/inputmethod/TypeinkInputMethodService.kt`
- Modify: `android/app/src/main/res/layout/activity_settings.xml`
- Modify: `android/app/src/main/res/values/strings.xml`

**步骤：**
1. 给设置页补 3 类用户可理解的开关：
   - 是否启用智能判停；
   - 当前 VAD 引擎；
   - 当前本地草稿引擎与模型状态。
2. 参考 参考键盘 的思路加入“预热”机制：
   - App 启动时可预热 VAD；
   - 输入法拉起前或首次进入设置页后可预热 sherpa 模型。
3. 设置页必须能解释“当前到底走的是哪条路径”：
   - `TEN-VAD`
   - `Energy VAD fallback`
   - `Sherpa local draft`
   - `System draft fallback`
4. 先不做复杂下载中心，第一版只支持“项目内置模型”或“手工放入固定目录”。

**完成标准：**
1. 用户能在设置页看懂当前是否真的启用了本地能力。
2. 首次录音不会因为临时加载模型而明显卡住。

### 阶段 E：最后再评估 WaveLineView，替换或吸收现有波形实现

**目标：** 在录音主链路稳定后，再决定是否用 `WaveLineView` 替换当前 `AudioWaveView`。

**Files:**
- Create: `android/app/src/main/java/com/typeink/ui/wave/WaveformView.kt`
- Modify: `android/app/src/main/java/com/typeink/prototype/AudioWaveView.kt`
- Modify: `android/app/src/main/java/com/typeink/inputmethod/ImmersiveEditorActivity.kt`
- Modify: `android/app/src/main/java/com/typeink/prototype/MainActivity.kt`
- Optional Create: `android/app/src/main/res/values/attrs_wave_line_view.xml`

**步骤：**
1. 先不要直接删 `AudioWaveView`，而是做 A/B 验证：
   - 当前自绘版是否已经足够；
   - `WaveLineView` 是否真的更顺滑、更省心。
2. 如果接 `WaveLineView`：
   - 先做一层 `WaveformView` 包装，不要让业务层直接依赖第三方控件 API；
   - 保持外部仍使用 `start / stop / updateAmplitude / setColor` 这一层。
3. 如果现有 `AudioWaveView` 已满足需求，就只借鉴它的灵敏度映射和生命周期处理，不强行替换。

**完成标准：**
1. 波形替换不影响识别与停录行为。
2. 若收益只在“更好看”而无明显稳定性提升，可明确放弃替换。

### 阶段 F：回归验证与文档收口

**目标：** 确保升级不是“只在演示页工作”，而是能在真实输入法场景里成立。

- Modify: `docs/安卓真机联调清单.md`

**步骤：**

**完成标准：**

2. 真机验证结论可重复，不依赖单一测试页面。

## 三、实施顺序建议

1. 先做阶段 A + B，不碰 sherpa。
2. `TEN-VAD` 真机稳定后，再开阶段 C + D。
3. `WaveLineView` 只有在前两步稳定后才进入评估。

## 四、暂不做的事

1. 暂不处理 icon 体系统一。
2. 暂不一口气支持多个 sherpa 本地模型。
3. 暂不做模型在线下载中心。
4. 暂不把“本地 ASR + 本地润色 + Compose 重构”绑成一轮。

## 五、主要风险

1. `sherpa-onnx` 会显著增加包体、资源管理和模型版本维护成本。
2. `TEN-VAD` 接入 native / AAR 后，需要额外注意 ABI、首次加载延迟和崩溃兜底。
3. 如果不先抽象接口，后面同时改 VAD、草稿识别、UI，回归成本会非常高。

## 六、推荐执行路径

最推荐的现实路线是：

1. 先做 `TEN-VAD`，把自动停录从“能量猜测”升级成“模型判断”；
2. 再做单模型版 `sherpa-onnx` 本地草稿；
3. 最后再评估 `WaveLineView` 是否值得替换当前波形。
