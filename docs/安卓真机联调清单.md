# 安卓真机联调清单

本文档是给后续 AI 编程助手和工程接手人用的“真机排查脚本”。
目的不是讲 Android 基础，而是避免每次装包、联调、报错时都重新猜“到底应该先查哪里”。

注意：

- 本文档里部分历史记录仍提到 `ws://IP:8000/ws`；
- 但当前仓库已移除那条 Python / Web 联调资产，Android 主线优先走“Android 直连 DashScope”；
- 如果你当前排查的是输入法 APK，请先优先参考：
  - [安卓输入法完整实现方案与测试计划.md](/Users/sam/Downloads/typeink/docs/安卓输入法完整实现方案与测试计划.md)
  - [AI助手-安卓输入法协作规则.md](/Users/sam/Downloads/typeink/docs/AI助手-安卓输入法协作规则.md)

## 1. 使用时机

只要涉及下面任一情况，就应该先按本清单走一遍：

1. 新打了 APK；
2. 改了 Android 客户端交互；
3. 改了 WebSocket 协议；
4. 改了 ASR / 重写器 provider；
5. 用户反馈“安卓端又不对了”。

## 2. 联调前先确认当前目标

真机联调前，先明确这次到底在验证什么。

不要一次混着测所有东西。

推荐在开始前先写清楚本次目标属于哪一类：

1. 连接验证
2. 实时草稿验证
3. 句末结果验证
4. 语音修改验证
5. `natural / formal` 风格验证
6. 修最近一句验证
7. 基础键盘按键验证
8. 崩溃与稳定性验证

## 3. 基础环境检查

### 3.1 APK 是否为最新包

默认检查这里：

- [android/app/build/outputs/apk/debug/app-debug.apk](/Users/sam/Downloads/typeink/android/app/build/outputs/apk/debug/app-debug.apk)

如果用户手机里已有旧包，建议：

1. 先卸载旧版；
2. 再安装新版；
3. 再开始测录音、识别和写回。

### 3.2 Android 本地配置是否就绪

当前主线优先检查：

1. `android/local.properties` 是否存在；
2. 里面是否包含 `typeink.dashscope.apiKey=...`；
3. 改完配置后是否重新构建过 APK。

注意：

- 根目录 `.env.local` 不再是当前 Android 主配置入口；
- 不要要求用户把真实 Key 写进源码或 README。

### 3.3 当前主线不再依赖本仓 Python 服务端

如果你还看到下面这些说法，请把它理解成历史路径，而不是当前主线要求：

- `ws://电脑局域网IP:8000/ws`
- `8001` 上游 ASR
- `uvicorn main:app`
- `source .env.local`

这些联调脚本对应的仓库资产已经删除，当前排查不要再沿这条路走。

### 3.4 首次安装后的基础检查

必须确认：

1. App 能正常打开；
2. 麦克风权限已授予；
3. 输入法已启用并已切换到 Typeink；
4. 没有出现 “DashScope Key 还没配置” 这类明显配置错误。

## 4. 首次打开 App 必查项

首次进 App，先看这几件事：

1. 页面能否正常打开；
2. 顶部是否显示当前后端地址；
3. 连接状态是否显示“已连接”；
4. 点击“重新连接”有没有明显报错；
5. 麦克风权限是否已授予。

如果这里就不正常，先不要继续测语音整理效果。

## 5. 核心场景测试顺序

推荐永远按下面顺序测。

## 场景 A：连接与会话建立

动作：

1. 打开 App；
2. 检查状态栏；
3. 点击“重新连接”。

预期：

1. 不白屏；
2. 不闪退；
3. 状态更新为已连接；
4. 服务端日志能看到连接建立。

## 场景 B：普通输入

动作：

1. 保证主文本框为空；
2. 不选中文本；
3. 点“开始录音”；
4. 说一句清晰中文；
5. 停止录音或等待句末。

预期：

1. 主文本框应出现实时草稿或云端 partial；
2. ASR 区应显示实时识别；
3. AI 区应在后续出现整理结果；
4. 最终主文本框应写回最终结果。

重点观察：

1. 首字是否出现太慢；
2. 主文本框是不是完全不动；
3. `asr_partial` 与 `llm_completed` 是否相互覆盖得很乱。

## 场景 C：语音修改

动作：

1. 先在主文本框放入一段已有文字；
2. 选中其中一部分；
3. 点“开始录音”；
4. 说出修改意图。

预期：

1. 客户端进入“语音修改”路径；
2. 服务端能拿到：
   - `selected_text`
   - `pre_text`
   - `post_text`
3. 最终结果应写回主文本框；
4. 没选中文本时不应误走修改模式。

## 场景 D：风格模式

动作：

1. 用同一段语音分别测 `保留原话` 和 `正式发布`；
2. 对比两次最终文本。

预期：

1. `保留原话`：
   - 去掉明显赘词
   - 修顺病句
   - 但仍像用户本人在说
2. `正式发布`：
   - 更像可直接发出去的书面表达
   - 更清晰、更完整、更有结构

如果两种结果几乎没有差异，要继续检查 prompt 或 style 透传是否失效。

## 6. 常见异常与第一判断

### 6.1 网页/页面打不开

先查：

1. 这是不是旧的 WebView 版本；
2. 当前装的是不是最新原生客户端 APK；
3. 手机里有没有残留旧包。

### 6.2 主文本框完全没有实时显示

先查：

1. 本地 `SpeechRecognizer` 是否工作；
2. 云端有没有 `asr_partial`；
3. 是否只在句末才收到 `asr_final`；
4. 手机机型是否对 `AudioRecord + SpeechRecognizer` 组合不友好。

### 6.3 结果像假文案

如果看到：

- “累计收到 xxx bytes 音频”

直接优先查后端 provider，别先改 Android UI。

### 6.4 连接状态显示已连接，但没有结果

先查：

1. 服务端是否收到音频二进制帧；
2. opcode 是否正确；
3. 主服务是否能连上 `8001` 上游；
4. 上游 ASR 是否有报错。

### 6.5 `formal` 效果不够像专业整理稿

先查：

1. 当前风格是不是确实切到了 `formal`；
2. style_mode 有没有正确透传到后端；
3. DashScope rewriter 是否真的被调用；
4. prompt 是否还需要继续增强。

## 8. 这份清单后续必须怎么维护

后续 AI 编程助手只要改了下面任一内容，就必须同步更新本清单：

1. Android 页面结构；
2. 录音按钮或交互流程；
3. 本地实时草稿逻辑；
4. WebSocket 连接方式；
5. 后端启动方式；
6. 真机排查优先级。

最低更新要求：

1. 核心测试顺序有没有变；
2. 新增了哪些必测场景；
3. 常见异常判断有没有变；
4. 当前最容易踩坑的点是什么。
