# 公开仓库与云效部署说明

本文给当前仓库（`https://github.com/shenyangs/typeink`）做一份最小操作说明，目标是：

1. 当前仓库保持 public 也能安全上传；
2. 后续随时切 private / rotate key 都能快速处理；
3. 使用 `GitHub + 云效 Flow + ECS` 时，减少配置歧义。

## 一、密钥位置（后续可随时拿掉）

当前约定：

- 本地私有密钥：`.env.local`（不进 Git）
- 对外示例模板：`.env.local.example`（可进 Git）

本项目中常见密钥变量：

- `DASHSCOPE_API_KEY`
- `TYPEINK_DASHSCOPE_API_KEY`

处理原则：

1. 不在代码里硬编码真实密钥；
2. 不在 README 中粘贴真实密钥；
3. 需要更换密钥时，只改 `.env.local`，无需改业务代码。

## 二、公开仓库发布前检查

每次推送前运行：

```bash
bash tools/check_public_release.sh
```

脚本会检查：

1. `.env.local` 是否被 Git 跟踪；
2. 仓库里是否出现疑似 `sk-...` 真实密钥；
3. `.env.local.example` 是否存在。

## 三、GitHub + 云效 Flow + ECS 最小流程

1. 代码托管在 GitHub（public/private 都可）。
2. 云效 Flow 添加代码源为 GitHub 仓库。
3. 在云效“服务连接”里配置 GitHub 授权（账号或令牌，需有仓库读取权限）。
4. 流水线构建后通过 SSH/Agent 部署到 ECS。
5. ECS 上由 `systemd` 管理两个进程：
   - `tools.dashscope_realtime_upstream:app`
   - `main:app`
6. Nginx 提供公网入口并启用 WebSocket 转发与 HTTPS。

## 四、为什么要两个后端进程

当前主服务的 ASR 走 `remote_ws` 模式，依赖上游 ASR WebSocket 服务，因此生产环境至少需要：

1. 上游 ASR 进程（接 DashScope 实时识别）
2. 主网关进程（对外提供 `/ws`、`/web-debug`、`/healthz`）

如果只启动主网关，会出现无法连接上游 ASR 的问题。
